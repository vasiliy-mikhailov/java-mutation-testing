# AGENTS.md

**What this is.** A research project whose **product is a portable Agent Skill bundle** ‚Äî `detect-java-version` + `detect-unit-testing-framework` + `improve-java-tests` (one `SKILL.md` each, markdown + YAML frontmatter) ‚Äî that a coding agent loads under **OpenHands (primary), opencode, or kilocode** and follows to **raise a Java repo's PIT mutation kill-rate**: detect the JDK the project needs, then find code the tests run but don't verify (or a class with no test at all), strengthen the existing test or write a brand-new `<Class>Test` from scratch, add tests that make the suite detect the surviving mutants under that JDK, keep only green improvements, open a PR. The harness triggers the skill; the host agent does the work with **standard tools only** (the build's PIT plugin, the matching JDK, `git`, `gh`) ‚Äî the skill is instructions, not a program we ship. Everything else in this repo exists to *forge and score* that skill, not to be shipped.

**Access & host.** Compute is `ssh mh` (user vmihaylov). Project at `~/improve-java-tests`. GitHub via the `gh` CLI. Inference (the panel's fixed model + the strong-rung reference driver) is the OpenAI-compatible Qwen FP8 endpoint; key in `.env` (`OC_KEY`/`QWEN_API_KEY`), read at runtime, never hardcoded. Maven/Gradle resolve through the on-host **Nexus** mirror on the `mvn-cache` docker network with warm cache volumes. Observability is the external frog's eye: write logs under `/var/log/observe/app/ijt/`, read the digest at `~/observe/{fast,medium,slow}.jsonl`.

1. **Problem (writing this file):** keep this file compact and outcome-named.
   - **Value:** short, outcome-named problems force the agent to re-derive the *how* from its tools each pass; the file rots when it accretes mechanism the agent could fill itself.
   - **Contract and constraints:** (operator-only; the agent does not edit this section). A problem is a self-amplifying attractor: one concern with a single extremum (its Reward) and a single trigger (its Attention mechanism), written for an intelligent agent searching a fuzzy environment ‚Äî supply no detail the agent can fill itself. Five sections each: Value, Contract and constraints, Solution search approach and hints, Reward, Attention mechanism. Backtick-pinned concretes (paths, command shapes, names) survive trims; aging enumerations (versions, model ids) get stripped. The one-liner names what the problem produces, not how. One problem is in foreground at a time; an interrupted problem is resumed, not restarted.
   - **Solution search approach and hints:** for each clause ask "why is this here?" ‚Äî strip mechanism the agent re-derives, keep what scopes when a rule applies. Re-audit a problem's agent-mutable sections after the operator touches its Contract.
   - **Reward:** cuts that lose words without losing a rule or its scope.
   - **Attention mechanism:** an operator edit to a Contract, or a problem visibly bloating, pulls this problem.

2. **Problem (no pre-defense):** meet a problem when it occurs, not in advance.
   - **Value:** speculative armor ‚Äî a guard, null-check, retry, fallback, or abstraction for a failure that has NOT happened ‚Äî is dead weight that hides where the code really breaks and multiplies paths no test exercises. Code stays smallest and most honest when every defense answers a failure actually seen. (Seneca: we suffer more in imagination than in reality.)
   - **Contract and constraints:** *(operator-only)* do NOT pre-defend. Write the direct path; let an unhandled case actually OCCUR ‚Äî a crash, a real trace ‚Äî before you guard it, then fix its ROOT (P3), not a blanket mask. No "just in case" null-checks/defaults/retries for inputs that cannot yet arise; no abstraction for a second caller that doesn't exist; no config knob nobody asked for. The bar to ADD a defense is a REPRODUCED failure, not a hypothesis. This is also why we do not pre-LIMIT: a cap, a short timeout, or an output truncation pre-arms against a hypothetical and in practice strangles the real path ‚Äî set timeouts effectively to infinity (a year; `AGENT_BACKSTOP` ~100y, stall ~1y, PIT/sandbox 1y via inner coreutils), ALWAYS send the FULL data never a truncated tail, and never cap size/depth/reasoning. A limit is a guard you have not earned (the `out[-3000:]` `log_tail` tail once starved the compile-gate of the `COMPILATION ERROR` marker it needed ‚Äî Watsonx, 2026-06-21). Complement of P3: P2 is BEFORE the failure (don't pre-arm), P3 is AFTER (root-cause, don't mask). Single carve-out: isolating a transient EXTERNAL fault at a boundary so it can't destroy unrelated work, and hardening an OBSERVED failure class (TCP keepalive + stall-detection after a real CLOSE_WAIT hang) is met-when-it-occurred, not pre-defense.
   - **Solution search approach and hints:** if you can't name the trace a guard prevents, delete the guard. A diff that GROWS to handle the imagined is the smell; the direct path is usually smaller.
   - **Reward:** the direct path shipped; zero guards without a matching observed failure.
   - **Attention mechanism:** a "just in case" comment; a null-check on a value that can't be null yet; a retry/fallback for a path that has never failed; an abstraction with a single caller "for the future"; a truncated tail (`[-N:]`) where full data should flow.

3. **Problem (bug fixing):** reproduce, root-cause, fix the cause, rerun clean ‚Äî never a workaround.
   - **Value:** a bug fixed at its root removes a whole failure class and leaves the code smaller; a guard/special-case hides the cause and accretes cruft. The discipline keeps the harness honest enough to trust its own measurements.
   - **Contract and constraints:** *(operator-only)* when something breaks ‚Äî (1) REPRODUCE (a real trace); (2) find the ROOT mechanism, not the symptom; (3) fix the ROOT so the error cannot occur ‚Äî NOT a `try/except`, "treat as X on failure", retry, or cap that lets the broken path keep running; (4) RERUN CLEAN; (5) clean up the consequences. A workaround is acceptable only as a labelled, tracked stopgap. **One exception: boundary fault-isolation** ‚Äî catching a TRANSIENT EXTERNAL fault (a vLLM 400 on malformed tool-call JSON; a dead inference socket) at the agent boundary so it doesn't destroy unrelated work is tolerance, not masking. The test: does the `try/except` hide a root cause we could fix (forbidden) or isolate an external flake so good work survives (allowed)?
   - **Solution search approach and hints:** trace on the host before theorising. When a fix ADDS code (a guard/branch/retry), suspect you are masking; the root-cause fix usually REMOVES code (the redundant context, the wrong path).
   - **Reward:** less code and fewer workarounds after the fix than before; zero masks left in the resolved path; the rerun is clean.
   - **Attention mechanism:** the diff is the channel ‚Äî a `try/except` that swallows, a "default to X on failure", a retry around a deterministic error, or a cap that hides an overflow, each signals the symptom was treated, not the cause.

## Skill cluster

4. **Problem (the skill):** one portable SKILL.md that makes a host agent raise a single Java file's PIT mutation score.
   - **Value:** a green suite proves nothing about catching regressions; mutation score (killed/total) is the real strength signal, and line coverage that kills no mutants is code executed but unverified. The deliverable is a hand manual any agent under any of the three harnesses can follow ‚Äî so it must lean only on tools all three share, and must be good enough that the agent, not us, does the killing. Its scope is one file: the harness feeds it a single class-plus-test via the traversal cluster (P8-10), and the skill never walks the repo itself. That separation is what keeps it portable, so anyone can run it on a single file with no harness.
   - **Contract and constraints:** The artifact is a single `SKILL.md` (frontmatter `name` + a `description` that triggers on "improve mutation coverage / improve the mutation score / strengthen Java tests") plus the `.claude-plugin` manifest ‚Äî **no runtime dependency**: the skill ships pure-LLM, so a using environment (opencode / kilocode / CI / a bare editor) needs no Python and no install. The mergeability reward is computed by the agent's own line-counted judgment against the SKILL.md rubric ‚Äî the **same pure-LLM rubric in training and in production**, with no separate deterministic scorer anywhere. We dogfood exactly what ships: a divergent internal scorer would let the public rubric rot untested, so there is no `reward.py` inside or out. The procedure it encodes: **detect the JDK the project actually needs (`detect-java-version`) and run every build/test/PIT command under it** ‚Äî a too-new JDK crashes PIT's forked minion; detect the **build tool** (`pom.xml` ‚Üí `pitest-maven`, `build.gradle` ‚Üí `gradle-pitest-plugin`) AND the **unit-testing framework + version** (via the **`detect-unit-testing-framework`** skill ‚Äî *resolve the version through `${...}`/BOM/`junit.version` property indirection*, confirming with the build's resolved `junit-platform-commons` version; a plain `<junit.version>6.x</>` mis-read as JUnit 5 crashes the minion), then take the framework-specific PIT-wiring path: **JUnit 4** ‚Üí the bare `mutationCoverage` goal (no plugin); **JUnit 5** ‚Üí add `pitest-junit5-plugin` to the PIT plugin classpath; **JUnit 6** (versioning unified, platform == jupiter version) ‚Üí a current PIT + `pitest-junit5-plugin` plus a `junit-platform-launcher` pinned to the project's platform version so engine and launcher align (otherwise the minion dies with `OutputDirectoryCreator not available`); **TestNG** ‚Üí add `pitest-testng-plugin` (externalized from PIT, no longer built-in). Inject the PIT plugin into the project's main `<build>`, never a `<profile>` build. A minion crash (`UNKNOWN_ERROR`) is **usually wrong wiring** ‚Äî most often a **JUnit-6 project mis-detected as JUnit 5** (resolve the version, above), not a real instrumentation problem ‚Äî or, under a too-new JDK, test-instrumentation too old ‚Üí apply Mockito/ByteBuddy floors or `--add-opens`; run PIT scoped to **one** logic-dense class, either an already-tested class to strengthen or an untested one that baselines all-NO_COVERAGE for the agent to write a fresh `<Class>Test` (whole-repo mutation is too expensive). **For a HUGE class (20k+ LOC / many thousands of mutants ‚Äî real repos have these) the work splits into TWO ROLES.** OpenHands sub-agents are enabled in the panel runner (`register_builtins_agents()` + a custom `mutation-tester` type registered from `docker/subagents/` + `get_default_tools(enable_sub_agents=True)`). The **main agent is the TEST-MANAGER**: it lists the methods and delegates each method, ONE AT A TIME (sequential ‚Äî parallel sub-agents collide on the shared test file), to a fresh **`mutation-tester`** sub-agent; it never authors tests itself, and when all methods are done it runs one whole-class PIT to confirm the lift (the harness commits + opens the PR). Each **`mutation-tester`** scopes PIT to its one method (`-DexcludedMethods=` the rest incl `<init>`/`<clinit>`, quoted), reads that method's survivors, APPENDS tests that kill them, RUNS them, FIXES any breakage, and returns a short summary ‚Äî its OWN bounded context, so the class scales to any size. The manager MUST delegate to `mutation-tester` (NOT `code-explorer`/`bash-runner`, which only analyze ‚Äî a 2026-06-19 smoke proved that mistake: CollectionUtils ran 2.6h / 7.1M tokens and wrote ZERO net tests because every delegation was analysis-only, and the build was left non-compiling with no compile-gate). This is **chunking the work, not capping the model**: mutating the whole God-class in one context drowns the agent (45M tokens ‚Üí cut-off ‚Üí BROKE_BUILD, as `CollectionUtils` first proved). (Harness-side Python method-batching of PIT was tried and REVERTED ‚Äî fragile `<init>`/`<clinit>` shell-mangling, aggregate 85 vs 209; let the AGENT chunk, not python.) Never limit the agent; hand it tractable pieces. read each survivor (`file:line:mutator`) from the PIT report; add tests that make the suite detect it by asserting the **correct** behaviour; re-run PIT scoped to confirm the lift; keep the improvement **only if every originally-passing test still passes and no existing assertion was weakened** ‚Äî additions are **append-only**, an existing test is never edited; recognize equivalent mutants and set them aside; **(framing: the skill IMPROVES the mutation score by strengthening tests to DETECT what the suite misses ‚Äî not "killing"; PIT still labels a detected mutant KILLED, the tool's term)** then branch, commit the additions, and open a **private-mirror PR** (test-only diff) whose body states the **mutation score AND unit-test line coverage** before‚Üíafter. The score must rise from a STRONGER test, never a laxer one. The skill encodes a **DUAL reward**: (a) a **strength reward ‚Äî +1 per mutant that no longer survives after the new tests (`survived_before ‚àí survived_after`; e.g. 4000 survivors ‚Üí 10 = reward 3990)**, and (b) a **mergeability reward = `0.9 ^ (penalty)`** applied by the agent itself against the SKILL.md rubric (pure-LLM, identical in training and production ‚Äî no separate deterministic scorer, so the public skill is exactly what we run) over the seventeen machine-checkable test-quality rules of the SKILL.md ¬ß6 rubric (enumerated once in P12, the single source of truth, so the count cannot drift) ‚Äî **penalty = the total LINES of bad test code** (every quality rule weighted by the line-count of the offending added code: an offending `@Test` counts its whole body, a per-line wart its matching lines; a 15-line warty method tanks the reward, a one-line slip barely dents it), with only the two build-outcome rules ‚Äî green, mutation-improving ‚Äî kept binary at 1 since they are not line-countable ‚Äî because "avoid that wart" is inert advice unless breaking it COSTS reward. An in-skill **Ralph loop** drives BOTH: it re-runs scoped-PIT ‚Üí read-survivors ‚Üí add-tests ‚Üí re-score, then re-judges the test against the rubric and fixes every broken rule **without losing a kill**, iterating until a full pass removes no survivors (strength reward 0, only equivalent mutants left) AND the mergeability reward is **1.0**. A PR opens at reward 1.0 (or a single documented residual unremovable without dropping a kill), never at merely-green.
   - **Solution search approach and hints:** coverage first ‚Äî a high-line-coverage, low-mutation-score class is the richest target. Per survivor, read the mutated operator+line and write the minimal assertion that distinguishes original from mutant. Don't chase equivalent mutants ‚Äî recognize the no-observable-effect pattern and skip. The wall patterns the strong-rung reference (`src/`, Qwen-driven) hits are the raw material for the manual; promote a pattern into `SKILL.md` only once it recurs. Re-enter when a panel run shows the host agent misread an instruction.
   - **Reward:** two axes the loop maximizes together. **Strength: +1 for each mutant that no longer survives after the added tests** ‚Äî `survived_before ‚àí survived_after` (e.g. 4000 ‚Üí 10 ‚Üí reward 3990); fewer survivors ‚Üí higher. **Mergeability: `0.9 ^ (penalty)`** per produced test file (LLM-judged against the SKILL.md rubric ‚Äî same rubric in training and production, no separate scorer) ‚Äî penalty = total LINES of bad test code across the seventeen quality rules (green + mutation-improving kept binary); 1.0 = no wart; this gates the PR. Aggregate signals: the corpus mutation-score lift (survived% before‚Üíafter) and the panel PASS rate (a run PASSES iff score rises, all baseline-passing tests stay green, and the PR opens).
   - **Attention mechanism:** a panel run where an agent following `SKILL.md` fails to lift the score, weakens a test, or breaks the build pulls this problem to fix the manual.

5. **Problem (harden the skill):** OpenHands is the learning oracle.
   - **Value:** during skill learning one strong reference agent gives the fastest honest signal of whether the manual works at all ‚Äî if OpenHands following the skill cannot kill survivors, the skill is the weak link, not the agent. Running all three agents per edit is slow and conflates "the skill is weak" with "an agent is weak" ‚Äî separate concerns (see the agent-parity problem).
   - **Contract and constraints:** The learning loop runs **OpenHands only** (the primary/reference agent) against the frozen eval set, `SKILL.md` installed at `.openhands/skills/`, driving the fixed FP8 model. Score before/after is measured by `src/pit.py` (never self-reported); every OpenHands run logs its **full dialog** (messages, reasoning, tool calls, sub-agent delegations AND each sub-agent's turn-by-turn internals) to `corpus/dialogs/<slug>.jsonl` (callback stream) + `<slug>-tree/` (Conversation `persistence_dir`); **NO_BASELINE** (un-scoreable baseline) AND **AGENT_ERROR** (agent crashed mid-run on an LLM/infra error ‚Äî `agent_rc != 0` ‚Äî leaving a partial broken file; distinct from a genuine rc=0 BROKE_BUILD where the agent finished but left the suite red), **MINION_CRASH** (PIT forked-minion crash) AND **NO_SURVIVORS** (baseline already saturated: 0 survivors, class marked in `saturated.txt`, no agent run) are **infra, excluded** from the rate, never counted as a skill FAIL. A skill edit is admitted **only if OpenHands' eval pass-rate does not regress** vs the pre-edit baseline; on regression the edit is reverted from backup. Gate each verdict on the result file MTIME **advancing**, not its existence; never hand-flip a verdict ‚Äî only a real re-run updates it. **Never set budgets or limits on the agent ‚Äî this is RLVR.** A model told it has a budget (iteration cap, wall-clock cap, token/turn budget, or any "your turns are limited" prompt) does NOT return a smaller honest result ‚Äî it **gives up and returns random noise** (NO_GAIN/BROKE_BUILD with 0 tests landed). Limits never trade depth for speed; they trade a real result for garbage. So run iterations, wall-clock, and turns **unbounded**, let the agent finish via its own stop action, and never communicate any budget to it ‚Äî the goal is to go **as deep as possible** on each target. There is NO timeout->kill: every work-path timeout is >= 1 year (agent backstop ~100y, stall-detection ~1y by default, PIT/sandbox bounded only by the inner coreutils `timeout`), so a productive run is NEVER guillotined. Treat the host as WHITEBOX: a genuine stall is a SPECIMEN for **anomaly -> investigate -> fix**, not a casualty of a blind deadline - the long timeout is exactly what let a 2026-06-21 dig pin a litellm hang on a half-closed CLOSE_WAIT socket (a short timeout would have killed+retried and hidden the bug forever). A DEAD inference connection is caught NOT by a request deadline but by **TCP keepalive** (`panel_oh_run.py` sets SO_KEEPALIVE on every socket, idle 60s then 15s x4 ~= 120s to detect) -> read error -> `num_retries` reconnects; a SLOW-but-alive response waits indefinitely. The dead-vs-slow distinction is the whole point: a timeout conflates them, keepalive separates them.
   - **Solution search approach and hints:** read the survivors OpenHands FAILS to kill (NO_GAIN/BROKE_BUILD) ‚Äî they name the skill's weak spots; sharpen the manual there. A starved/misconfigured OpenHands fakes a FAIL ‚Äî confirm `drop_params=False` + the starvation check before trusting it. Re-enter per skill edit.
   - **Reward:** OpenHands' eval pass-rate before vs after an edit; keep iff it does not regress.
   - **Attention mechanism:** an OpenHands pass-rate drop after a skill edit pulls this problem.

6. **Problem (agent parity):** bring opencode/kilocode up to OpenHands on the same skill.
   - **Value:** a skill hardened only against OpenHands may lean on its strength; the honest portability signal is unrelated off-the-shelf agents succeeding on the *identical* manual. But when OpenHands passes and another agent fails, the variable is the **agent**, not the skill ‚Äî so the fix is to tune the lagging agent, not rewrite the manual.
   - **Contract and constraints:** Periodically (NOT every skill edit) run **all three agents** ‚Äî OpenHands, opencode, kilocode ‚Äî on the **current** skill over the eval set, the agent the only variable (same skill, same FP8 model, same `pit.py` scoring). A target where **OpenHands PASSes but an off-the-shelf agent does NOT** (baseline scored; NO_BASELINE / AGENT_ERROR / MINION_CRASH / saturated-NO_SURVIVORS all excluded as infra or un-scoreable) is a **fine-tuning work item for THAT agent**: tune the agent's config / harness / scaffolding (a silent-401 no-op from a missing `apiKey: {env:OC_KEY}`, a starved budget, a wrapper-execution fumble) until it matches OpenHands ‚Äî **skill and model held fixed**. The skill is NOT edited here. Escalate to a skill clarification ONLY if the SAME gap hits multiple agents (then it is a manual ambiguity ‚Üí the learning problem).
   - **Solution search approach and hints:** diff per-target reference(OpenHands)-vs-agent verdicts; per gap read the lagging agent's transcript ‚Äî a tiny no-op log = a config/401 problem; an engaged-but-fumbled run = an execution limit (tune its scaffolding or accept it). Re-enter when a parity sweep shows OpenHands-pass / agent-fail targets.
   - **Reward:** per-agent parity gap (count of OpenHands-pass-but-agent-fail targets) shrinking toward zero.
   - **Attention mechanism:** a parity sweep where OpenHands passes a target an off-the-shelf agent fails pulls this problem.

7. **Problem (candidate corpus):** top-starred, maintained, compilable, test-bearing, mutatable.
   - **Value:** the panel needs a stable set of green, test-bearing repos with a non-trivial target; feeding repos that don't compile, have no tests, or only do CRUD wastes the expensive runs. Targeting the **HIGHEST-STARRED** maintained repos (not mid-size) makes each PASS's eventual upstream merge a stronger adoption signal (P12) ‚Äî a merge into a popular repo carries the skill further.
   - **Contract and constraints:** Discovery is **maintained + top-stars** ‚Äî rank Java repos by **stars descending** and take the most popular maintained, test-bearing ones at HEAD (`gh search repos --language=java --sort=stars`, recent `--updated>=` = maintained; **multi-module is fine** ‚Äî route PIT to the module that owns the candidate class; exclude only what the skill cannot run on: Android and demo/example/tutorial names). A **size cap is a BUILD-COST guard (P11), not a popularity filter** ‚Äî it keeps out only pathological mega-monorepos the sandbox cannot build cheaply. One cheap recursive-tree probe before any clone. Shallow-clone the **current** commit (not a historical walk). Gate cheapest-first **under the repo's detected JDK** (`jdkdetect`): (1) compiles; (2) static `@Test` count ‚â• threshold; (3) tests green; (4) **PIT can baseline at least one TESTED candidate** (the probe tries every tested candidate in rank order, skipping untested ones since they baseline trivially at 0 coverage and prove nothing about the minion, and admits on the first that baselines; a repo whose minion crashes on all is pre-filtered OUT). A repo whose tests pair to NO main is rejected `no_tested_target` before the build, since candidate_classes emits every main so `cands` is non-empty even with zero tested classes. `candidate_classes` is **every main class** in the repo, not only test-paired ones: each carries its owning module and a test target: the existing `FooTest` when one exists, else the conventional `<pkg>.<Class>Test` the agent will CREATE (PIT baselines an untested class as all-NO_COVERAGE, killed 0, so a no-test class is a first-class target, and per P12 the most valuable one). An admitted record `{repo, sha, build_tool, jdk, repo_dir, module, modules, test_count, candidate_classes, probe_class, probe_score}` (each candidate carries its `module`, `has_test`, `n_test`, and src/test paths) persists to `corpus/queue.jsonl`, deduped per repo against a **seen-set** (every repo gated, admitted OR dropped, is never re-gated). The dig runs **perpetually** (no target). Junk never enters.
   - **Solution search approach and hints:** reject fast on no-tests / won't-compile / red before any costly run. The survivor count is the filter, not a static heuristic: take every main class and let PIT's baseline sort them: zero survivors (a trivial getter, a saturated suite) is a no-agent skip, many survivors (an untested logic-dense class) is the target. Re-enter when the corpus runs low.
   - **Reward:** hit-rate (discovered ‚Üí admitted green + test-bearing + mutatable) and the supply of un-scored targets.
   - **Attention mechanism:** the queue file ‚Äî a fresh admitted target pulls P5; a gate failure drops the repo.

## Traversal cluster

8. **Problem (improve-java-tests-for-repo):** raise a whole repo's mutation coverage by decomposing it top-down, never in one prompt.
   - **Value:** a repo, above all a multi-module one, never fits one agent context, so no single prompt can mutation-test it; the only tractable path is repo ‚Üí module ‚Üí file, applying the single-file skill (P4) to each file in its own bounded context. The loop scales the work to any repo size and keeps the leaf skill portable: the file level is the public skill, runnable on one file by anyone without this harness.
   - **Contract and constraints:** *(operator-only)* draw the ranked flat file-targets (`draw.draw`, all admitted candidates, uncapped) and regroup them by repo into `repo -> {module -> files}`, then submit ONE worker per REPO (`_run_repo`): a lane owns a whole repo and walks it module by module, file by file, against a single clone it makes ONCE (so a multi-module repo is cloned once, not once per module). IMPROVE_WORKERS repos run in parallel. Module enumeration comes from the corpus candidate records (the gate's per-candidate `module` + the record's ranked `modules`), not a run-time build-file walk. A repo is a container of modules; a flat single-module repo is just the one-module case.
   - **Solution search approach and hints:** the module is already on each candidate (gate's per-candidate `module`, and the record's ranked `modules`); group by it preserving draw rank. Do not re-read root `<modules>`/`settings.gradle` at run time.
   - **Reward:** every mutatable file in the repo reached by the skill over cycles, no module skipped for any reason other than it does not build.
   - **Attention mechanism:** a repo admitted to the corpus (P7), or a whole-repo run requested.

9. **Problem (walk-modules):** iterate a repo's modules, building each once before descending.
   - **Value:** the module is the build + convention unit: PIT compiles per module (`mvn -pl <module>`), test idiom is per module, and a module that does not build in isolation makes every class inside it burn a separate rebuild (the NO_BASELINE bleed seen on truth / nexus / optaplanner). Building the module once, then skipping it wholesale when it does not build, stops that waste and holds one convention context for every file under it.
   - **Contract and constraints:** *(operator-only)* for each module in the repo's SINGLE shared clone (P8 clones the repo once per lane, modules are not re-cloned), detect its JDK / build tool / framework, and BUILD it once (`-pl <module> -am -DskipTests install`), a compile+install, NOT a PIT run: the PIT baseline is per-file in walk-files (P10). If it does not compile, skip that module this cycle, and PERMANENTLY blacklist it (module-keyed `no_build`, key `<repo>#<module>`) ONLY on a deterministic COMPILATION ERROR; a transient resolver/network/OOM flake (Nexus 403/5xx, dead group member, container eviction) retries next cycle, so one bad infra night never shrinks the admitted corpus. **Fair scheduling is per-repo (P8):** the repo lane processes at most `REPO_FILE_BUDGET` (40) files per cycle ACROSS all its modules then yields, so a giant repo re-competes instead of pinning a worker; deferred files re-enter the next draw. **Gradle is the exception:** its pitest task must be the first gradle invocation (a prior build poisons the config-cache), so gradle files skip build-once and use their own fresh per-file clone (the improve loop draws maven only).
   - **Solution search approach and hints:** one module build serves all its files; keep per-file isolation in the shared clone by committing the post-build state as a `ijt-baseline` commit, then `git reset --hard` + `git clean -e **/target` before each file (drops the prior agent's added tests, keeps the compiled build).
   - **Reward:** each building module BUILT exactly once (no per-class rebuilds), every non-building module skipped and blacklisted only on a real compile error.
   - **Attention mechanism:** a module handed down by improve-java-tests-for-repo (P8), or a NO_BASELINE cluster that traces to per-class rebuilding.

10. **Problem (walk-files):** apply the single-file skill to each candidate file of a built module.
   - **Value:** the file is the unit of skill-application: PIT scopes to one class and one agent context holds one class, with method-level delegation inside for a huge one (P4). Iterating an already-built module's files reuses its compile and its idiom, so each file costs only its own PIT and test-writing.
   - **Contract and constraints:** *(operator-only)* for each candidate file in the built module (test-paired OR untested), baseline it first; if the baseline has ZERO survivors (a trivial zero-mutant class, or a suite that already kills everything) mark it saturated and skip it with NO agent (PIT's survivor count is the filter), so every class is admitted but only survivor-bearing ones cost an agent run. For a survivor-bearing file spawn a fresh agent that applies the improve-java-tests skill (P4) to that one file, either strengthening its existing test or CREATING `<Class>Test` following the module's idiom when the class has none, then a disinterested judge sub-agent scores the added tests against the mergeability rubric, reading one sibling test in this same module for idiom (P12 rules 16-17). One file per agent, fresh context each; files within a module are independent, so a failure on one never blocks the next. This walk is exactly the public skill invoked once per file: hand the skill a single file and it runs anywhere; this harness only chooses which file and when.
   - **Solution search approach and hints:** the writer never scores its own tests (a self-score is a conflict of interest, P12); a separate judge reads the module's sibling test the writer's module context already surfaced. Independent files may run concurrently only when they do not share a test file.
   - **Reward:** every candidate file in the module improved by a green, mutation-raising, judge-cleared diff, or explicitly left alone when already saturated.
   - **Attention mechanism:** a built module handed down by walk-modules (P9).

## Substrate cluster

11. **Problem (substrate):** docker-bounded, reproducible, observable.
   - **Value:** a score must vary only with the skill and the code under it; an unstable toolchain or a full disk corrupts the comparison, and a blind stalled run hides itself.
   - **Contract and constraints:** Project at `~/improve-java-tests` (the one pinned path). Everything runs in containers: a **minimal sandbox image per LTS JDK** (`java-{8,11,17,21,25}-improve-java-tests-sandbox`, Maven via Nexus) for build+PIT ‚Äî **NO scripts or config are baked in; the Nexus `settings.xml`, the `jrun` helper, and all pipeline code are bind-mounted at runtime, so editing them never triggers a rebuild**. Each repo is routed to the image matching its **detected JDK**. A `ijt-orch` image (python + docker CLI + `gh`) spawns sandbox siblings via the mounted socket; the per-harness panel image likewise mounts the socket + `jrun` so the **agent runs its own build/test/PIT in the proper LTS sandbox** (`jrun <jdk> '<cmd>'`). Bind-mount data at its real host path so paths resolve across the socket. Bound every build/PIT run with an **inner** timeout (a host-side kill drops only the docker client). Pin Maven/Gradle caches.    - **Reward:** zero disk-full crashes, reproducible per-run wall-clock, docker-bounded everything.
   - **Attention mechanism:** host metrics + the frog's-eye digest ‚Äî disk creeping full, or a run gone silent.

## Adoption cluster

12. **Problem (offer mergeable tests upstream):** turn high-lift panel PASSes into test-only PRs to receptive maintainers ‚Äî quality-filtered, demand-targeted, merge-measured.
   - **Value:** the corpus PASS rate proves the skill in vitro; a MERGED test-only PR is ground-truth adoption AND a real-world distribution channel (the disclosed mutation-testing methodology travelling on every merged PR). But an unsolicited or wart-ridden PR is NEGATIVE advertising carrying the skill's name ‚Äî so this problem gates hard on quality and receptiveness, kept distinct from the in-vitro learning oracle (P5) and the private-mirror measurement of the skill (P4).
   - **Contract and constraints:** *(operator-only)* a PR opens ONLY after three gates all pass. (1) **Quality filter = mergeability reward 1.0** ‚Äî avoid costs nothing if not rewarded, so the wart catalog is encoded as the seventeen machine-checkable rules of the `improve-java-tests` SKILL.md rubric, line-counted by a disinterested judge sub-agent, never the writer that produced the tests (a self-score is a conflict of interest: the writer is driven to finish and inflates it) (pure-LLM, no runtime; judged identically in training and production, no separate deterministic scorer) (**`reward = 0.9 ^ penalty`**, penalty = total LINES of bad added test code; green + mutation-improving kept binary at 1): api-only (no reflection-into-internals / implementation-coupled assertions), every-test-asserts (no zero-assertion / coverage-only), no-vacuous-assert, no-adnt-only, deterministic, no-disabled, no-unused-code (unused import / private field / private method ‚Äî added after DSpace/xoai#133), additive-only, green (invented/nonexistent API fails to compile here), mutation-improving, no-partial-assert (‚â•2 substring checks on one variable ‚Äî added after zowe #524), no-trivial-accessor-test (pure getter/setter/equals/hashCode/toString tests = maintainer noise ‚Äî added after spring-ai-watsonx #145), no-inner-class (no nested / @Nested / helper class declared in the test ‚Äî keep tests flat), no-comment-spam (standalone comment lines may not out-pace code more than 1 per 4 code lines ‚Äî comment why not what), no-tooling-exhaust-comments (added after apache/tika #2917: test comments must not name PIT mutant operators, say "kills the surviving mutant", or hardcode production line numbers; state the behaviour under test), no-thin-delegator (test where the logic lives, never a trivial wrapper whose behaviour is really in another class; added after assertj #4310), repo-idiomatic (match the target's CONTRIBUTING conventions and its module's existing test idiom, e.g. assertj's given/when/then; added after assertj #4310). To score the last two the judge first reads the target's CONTRIBUTING and one sibling test in the same module, so a green compile plus checkstyle and spotless is necessary but not sufficient: assertj #4310 built clean yet was closed for missing given/when/then and for testing a trivial wrapper. Every quality rule is weighted by the line-count of the offending code (whole-line-weighting generalizes the original per-line unused-code penalty), so warty bulk costs proportionally. Each maintainer reject mechanizes into a new rule (P12‚ÜíP5). A PR opens ONLY at **reward 1.0** (or a single documented residual unremovable without dropping a kill); any wart drops below 1.0 and the result returns to P5. The PR carries only tests that clear every rule, never raw agent output ‚Äî same leave-it-mergeable root-cause discipline as P3. (2) **Demand-targeted** ‚Äî PR only where the maintainer signals they WANT tests: an open "help wanted: tests / improve coverage" issue (`resolves #N`) or an actively-maintained repo with a low-coverage target; never spray strangers. (3) **Start small + measure** ‚Äî open a handful of the highest-lift results (big coverage + mutation-score lift, clean small diff) first and widen only as merges land; the metric is MERGE rate, never open rate. Every PR is test-only, no production code touched, in one of two shapes: append-only additions to an existing test, or a brand-new `<Class>Test.java` for a previously-untested class (the most welcome contribution), opened operator-fork ‚Üí upstream ONLY with the operator's explicit go-ahead (never autonomously), honouring each repo's CONTRIBUTING / CLA / PR-title conventions, one PR per target with no churn; the body reports mutation score AND line coverage before‚Üíafter WITHOUT overselling (regression-strengthening tests that pin current behaviour, not bug-finding) and carries the mandatory **AI-pipeline disclosure footer** (never undisclosed AI use): a short `How this was produced` note that the PR was generated by an AI-assisted pipeline built on mutation testing: PIT mutates the class (flipping conditions / boundary cases) to surface edge cases the existing tests miss, a focused test is written for each surviving mutant, and PIT is rerun to confirm that test actually kills that specific mutant, so every added test is verified to catch a concrete edge case the suite missed before. Disclosure pre-empts the undisclosed-AI rejection class observed at RxJava/cucumber (P12‚ÜíP5 maintainer signal), and the methodology itself signals rigour rather than slop. A result that fails the filter is NOT PR'd ‚Äî it returns to P5 as a skill-weakness signal.
   - **Solution search approach and hints:** the filter's reject-list IS the wart catalog the panel digs surfaced ‚Äî reflection-into-`elementData` capacity tests, zero-assert coverage tests, duplicate / mis-resolved-overload tests, comments that overclaim what a test kills, invented API. A maintainer rejection ‚Äî or a "we don't want characterization tests" reply ‚Äî names exactly what to tighten; feed it to the filter. Re-enter on a high-lift PASS, or a maintainer / CI response on an open PR.
   - **Reward:** merged upstream test-only PRs (primary); the funnel panel-PASS ‚Üí filter-clean ‚Üí demand-matched ‚Üí opened ‚Üí merged, read as MERGE rate not open rate.
   - **Attention mechanism:** a high-lift panel PASS triggers the quality filter and, if clean with a receptive target, a PR; a maintainer reply or CI result on an open PR pulls this problem back; a filter reject publishes the wart to P5 as a skill gap.
